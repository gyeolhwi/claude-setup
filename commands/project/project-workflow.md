# Project Workflow Commands

`docs/` 폴더 기반 프로젝트에서 AI가 실행하는 워크플로우.
각 모드는 docs/ 내부 파일을 읽고, 처리하고, 업데이트한다.

## 전제 조건

- `project-docs` 스킬로 생성된 `docs/` 폴더가 프로젝트에 존재해야 한다
- 커맨드 실행 전 `docs/INDEX.md`를 먼저 읽어서 프로젝트 컨텍스트를 파악한다
  - **예외:** classify 모드와 `/project-docs-gen`은 `INDEX.md` 없이도 동작한다 (없으면 스킵)

## 모드 목록

아래는 `project-workflow` 내부의 서브모드다. 독립 슬래시 커맨드가 아니다.
자연어로 호출한다: "피드백 처리해줘", "개발 시작해줘", "리뷰해줘" 등.

| 모드 | 호출 예시 | 읽는 파일 | 쓰는 파일 |
|------|----------|-----------|-----------|
| feedback | "피드백 처리해줘" | `FEEDBACK.md` | `FEEDBACK.md` |
| issues | "이슈 확인해줘" | `05_issues/open/` | `05_issues/` |
| dev | "개발 시작해줘" | `CONVENTIONS.md`, `02_architecture/`, `01_specs/`, `FEEDBACK.md` | `FEEDBACK.md`, `04_tasks/` |
| design | "디자인 수정해줘" | `FEEDBACK.md`, `03_references/design/` | `FEEDBACK.md` |
| review | "리뷰해줘" | `CONVENTIONS.md` | `06_changelog/CHANGELOG.md` |
| sync | "동기화 체크해줘" | 전체 docs/ | — (보고만) |
| status | "현재 상태 알려줘" | 전체 docs/ | — (보고만) |
| classify | "분류해줘" | `99_inbox/` | `01_specs/`, `02_architecture/` 등 |

## feedback 모드 — 다건 피드백 일괄 처리

사용자가 `FEEDBACK.md`에 여러 수정 요청을 모아서 작성한 뒤, "피드백 처리해줘"라고 요청하면 일괄 처리한다.
간단한 단건 요청은 이 모드 없이 채팅으로 직접 말하면 된다.

### 실행 흐름

1. `docs/FEEDBACK.md` 파일을 읽는다
2. `상태: 대기` 인 피드백 항목을 모두 추출한다
3. 각 항목을 우선순위별로 정렬하여 보여준다 (높음 → 보통 → 낮음)
4. 사용자에게 어떤 피드백부터 처리할지 확인한다 (전체 일괄 처리도 가능)
5. 선택된 피드백을 처리한 후:
   - 해당 항목의 상태를 `대기` → `완료`로 변경
   - "리뷰해줘"로 변경사항을 기록할 수 있습니다. 안내

### 출력 형식

```
📋 대기 중인 피드백 {n}건

[높음]
1. {제목} — {유형} — {대상}
   내용: {내용 요약}

[보통]
2. {제목} — {유형} — {대상}
   내용: {내용 요약}

어떤 피드백부터 처리할까요?
```

## issues 모드 — 이슈 확인 & 분석

`docs/05_issues/open/` 폴더의 이슈를 파악하고 분석한다.

### 실행 흐름

1. `docs/05_issues/open/` 폴더의 모든 `.md` 파일을 읽는다
2. 파일이 없으면 "현재 오픈 이슈가 없습니다"를 보여주고, `FEEDBACK.md`에서 이슈로 전환할 항목이 있는지 확인한다
3. 각 이슈를 요약하여 보여준다
4. 이슈 간 연관성이 있으면 분석하여 알려준다
5. 해결 방안을 제안한다
6. 이슈가 해결되면 `05_issues/open/` → `05_issues/resolved/`로 이동

### 이슈 파일 형식 (새로 생성 시)

파일명: `YYYY-MM-DD-{제목-kebab-case}.md`

```markdown
# {이슈 제목}

- **날짜**: YYYY-MM-DD
- **심각도**: `critical` | `major` | `minor`
- **상태**: `open` | `investigating` | `resolved`
- **관련 파일**: {파일 경로}

## 설명

{이슈 상세 설명}

## 재현 방법

{해당되는 경우}

## 해결 방안

{분석 후 작성}

## 해결 기록

{해결 과정 기록}
```

## dev 모드 — 기능 개발

피드백이나 직접 요청을 기반으로 기능을 개발한다.

### 실행 흐름

1. `docs/INDEX.md` → 프로젝트 컨텍스트 파악
2. `docs/CONVENTIONS.md` → 코드 스타일 확인
3. `docs/02_architecture/` → 관련 아키텍처 문서 확인
4. `docs/01_specs/` → 요구사항정의서, 기능명세서, API 명세서 등 확인 (있는 경우)
5. `docs/FEEDBACK.md` → 대기 중인 관련 피드백이 있으면 확인
6. 개발 범위를 사용자에게 보여주고 진행:
   - 어떤 기능인지
   - 영향 받는 파일 목록
   - 예상되는 변경사항
8. 완료 후 업데이트:
   - `FEEDBACK.md` 해당 항목 상태 → `완료`
   - `04_tasks/current-sprint.md` 업데이트
   - "리뷰해줘"로 변경사항을 기록할 수 있습니다. 안내

### 개발 전 체크리스트 (AI가 내부적으로 확인)

- [ ] CONVENTIONS.md 코드 스타일 확인했는가
- [ ] 기존 컴포넌트/유틸과 중복되는 것은 없는가
- [ ] architecture 문서와 일치하는 방향인가
- [ ] 영향 받는 다른 기능이 있는가

## design 모드 — 디자인 수정

UI/UX 관련 피드백을 처리한다.

### 실행 흐름

1. `docs/FEEDBACK.md`에서 유형이 `디자인`인 항목 확인
2. `docs/03_references/design/` 폴더에서 관련 레퍼런스 확인
3. 수정 범위를 사용자에게 시각적으로 설명:
   - 현재 상태 (가능하면 코드나 구조 설명)
   - 변경 후 예상 결과
4. 사용자 확인 후 수정 진행
5. 완료 후:
   - `FEEDBACK.md` 해당 항목 상태 → `완료`
   - "리뷰해줘"로 변경사항을 기록할 수 있습니다. 안내

### 디자인 수정 시 주의사항

- 기존 디자인 시스템/테마 변수를 우선 사용한다
- 새로운 색상, 폰트, 간격 추가 시 CONVENTIONS.md 확인
- 반응형 대응 여부를 사용자에게 확인한다
- 수정 전후를 명확하게 설명한다

## review 모드 — 변경사항 리뷰

코드 변경 후 리뷰와 문서 업데이트를 수행한다.

### 실행 흐름

1. 최근 변경된 파일들을 확인한다 (git diff 또는 사용자 설명)
2. `docs/CONVENTIONS.md` 기준으로 코드 리뷰:
   - 네이밍 규칙 준수 여부
   - 패턴 일관성
   - 에러 처리
   - 타입 안전성
3. 리뷰 결과를 보여준다
4. `docs/06_changelog/CHANGELOG.md` 업데이트
5. 필요시 architecture 문서 업데이트 제안

### 리뷰 출력 형식

```
🔍 코드 리뷰 결과

✅ 잘된 점
- {내용}

⚠️ 개선 제안
- {파일}: {내용}

📝 문서 업데이트 필요
- {어떤 문서에 무엇을 추가해야 하는지}
```

## sync 모드 — 문서 ↔ 코드 동기화

실제 코드와 docs 문서가 일치하는지 확인한다.

### 실행 흐름

1. `docs/` 내 문서 존재 여부와 폴더 구조 정합성 확인
   - `01_specs/`에 문서가 있는지 (비어있으면 "/project-docs-gen으로 생성하세요" 안내)
   - `02_architecture/`에 overview.md가 작성되어 있는지
2. `docs/02_architecture/` 문서에 기술된 구조와 실제 프로젝트 구조 비교
3. `docs/CONVENTIONS.md`의 규칙이 실제 코드에 반영되어 있는지 확인
4. `docs/04_tasks/current-sprint.md`에 완료된 항목 중 `completed/`로 안 옮겨진 것 확인
5. `docs/05_issues/open/`에 이미 해결된 이슈가 남아있는지 확인
6. 불일치 항목을 보고하고 자동 수정을 제안

### 출력 형식

```
🔄 동기화 체크 결과

✅ 일치 항목: {n}건
⚠️ 불일치 항목: {n}건

불일치 상세:
1. {문서} ↔ {실제}: {내용}
   → 제안: {자동 수정 방안}

자동으로 수정할까요?
```

## status 모드 — 프로젝트 현재 상태 요약

프로젝트 전체 상태를 한눈에 보여준다.

### 실행 흐름

1. 모든 docs/ 파일을 스캔
2. 요약 대시보드 생성

### 출력 형식

```
📊 프로젝트 상태: {프로젝트 이름}

📋 피드백: {대기 n건} | {진행중 n건} | {완료 n건}
🐛 이슈: {오픈 n건} | {해결 n건}
📝 태스크: {현재 스프린트 n건} | {백로그 n건}
📅 마지막 변경: {CHANGELOG 최신 날짜}

⚡ 즉시 처리 가능:
- {높은 우선순위 피드백이나 크리티컬 이슈 요약}
```

## classify 모드 — inbox/ 문서 분류

`docs/99_inbox/`에 있는 문서들을 내용을 분석하여 올바른 폴더로 이동한다.
`project-docs-gen`으로 생성된 문서들을 docs/ 구조에 편입시키는 커맨드다.

### 실행 흐름

1. `docs/99_inbox/` 폴더의 모든 `.md` 파일을 읽는다
2. 비어있으면 "분류할 문서가 없습니다"를 출력하고 종료한다
3. 각 파일 내용을 분석하여 아래 분류 기준에 따라 이동 대상 폴더를 결정한다:

| 문서 유형 | 이동 폴더 |
|----------|----------|
| 요구사항정의서, 기능명세서, 정보구조도 | `docs/01_specs/` |
| ERD 설계서, API 명세서, 기술 스택 결정서 | `docs/01_specs/` |
| 플로우차트, 와이어프레임, 시나리오 케이스, 역할/권한 매트릭스 | `docs/01_specs/` |
| 시스템 아키텍처, 구조 설계 문서 | `docs/02_architecture/` |
| UI/디자인 레퍼런스 | `docs/03_references/design/` |
| API 외부 문서, 라이브러리 설명 | `docs/03_references/api/` |
| 태스크, 백로그, 마일스톤 | `docs/04_tasks/` |
| 분류 불명확 | `docs/99_inbox/` 유지 후 수동 확인 목록 출력 |

4. 이동 계획을 사용자에게 보여주고 확인을 받는다
5. 확인 후 파일을 이동한다
6. `docs/INDEX.md`에 이동된 문서 링크를 추가한다

### 출력 형식

```
📥 99_inbox/ 문서 {n}건 발견

분류 계획:
- requirements.md → docs/01_specs/
- api.md → docs/01_specs/
- overview.md → docs/02_architecture/

이동을 진행할까요?
```

이동 완료 후:
```
✅ {n}개 문서 분류 완료
📁 docs/01_specs/: {n}개
📁 docs/02_architecture/: {n}개
⚠️ 수동 확인 필요: {파일 목록}
```

## 공통 규칙

### 모든 모드 실행 전

1. `docs/INDEX.md`를 먼저 읽어 프로젝트 컨텍스트를 파악한다 (없으면 스킵 — classify는 INDEX 없이 동작 가능)
2. 관련 문서가 있으면 반드시 읽고 나서 작업한다
3. `CONVENTIONS.md`가 존재하면 코드 생성 시 반드시 따른다

### 모든 모드 실행 후

1. 변경된 docs 파일이 있으면 어떤 파일이 변경되었는지 알려준다
2. 다음 추천 액션을 안내한다 (예: "리뷰해줘"로 변경사항을 기록할 수 있습니다)

> **CHANGELOG 기록은 review 모드만 담당한다.** 다른 모드에서 직접 CHANGELOG를 수정하지 않는다.

### 파일 수정 시

- 기존 내용을 지우지 않는다 (추가/수정만)
- 상태 변경은 인라인으로 한다 (`대기` → `완료`)
- 날짜는 항상 YYYY-MM-DD 형식
